# Tech Debt Story TD-18-22: Receipt Images Fail to Load in Image Viewer

Status: done

> **Source:** Production observation (2026-04-06)
> **Priority:** CRITICAL | **Estimated Effort:** 5 points

## Intent
**Epic Handle:** "One statement in, many verified transactions out"
**Story Handle:** "If I scanned it, I should be able to see it again"
**Value:** V5 — "Easier than the receipt drawer" — receipt images are the proof-of-purchase. If they disappear after scanning, users lose trust and fall back to keeping paper receipts.

## Story
As a **user**, I want **to view my scanned receipt images after saving a transaction**, so that **I can verify what was scanned and reference the original receipt**.

## Background

### Observed behavior (production, 2026-04-06)
1. Scan a receipt — scan completes, QuickSaveCard shows correct data
2. Save or edit the transaction
3. Open the transaction in the editor — thumbnail is visible
4. Tap the thumbnail to open full image viewer
5. Image viewer opens (dark overlay, X button in top right) but shows **"Failed to load image"** with an image icon placeholder

### Root cause: Image URLs point to deleted Storage files

The scan pipeline has a gap in the image lifecycle:

```
Upload: Client → pending_scans/{userId}/{scanId}/image_N.jpg  ✓
Process: CF reads pending images, saves result with same URLs  ✓
Save:    Transaction saved with imageUrls pointing to pending path  ✓
Cleanup: onPendingScanDeleted CF deletes pending Storage files  ✓
View:    Transaction.imageUrls → DELETED files → "Failed to load"  ✗
```

**`copyPendingToReceipts()` exists in `pendingScanUpload.ts:114` but is NEVER CALLED.** Images are never copied from the temporary `pending_scans/` path to the permanent `receipts/` path. When the pending scan document is cleaned up, `onPendingScanDeleted` calls `deletePendingScanImages()`, destroying the only copy.

### Why the thumbnail works
The `thumbnailUrl` is generated by the Cloud Function (`processReceiptScan.ts:323`) and stored at a permanent path in Storage (not under `pending_scans/`). It survives cleanup.

### Evidence in code
- `pendingScanUpload.ts:114` — `copyPendingToReceipts()` defined but 0 callers (dead export)
- `processReceiptScan.ts:333` — `imageUrls: data.imageUrls` stores pending path URLs as-is
- `onPendingScanDeleted.ts:50` — `deletePendingScanImages(userId, scanId)` deletes pending files
- `ImageViewer.tsx:140` — `src={images[currentIndex]}` loads from transaction.imageUrls (broken)
- `EditView.tsx:220-222` — passes `images={currentTransaction.imageUrls!}` to ImageViewer

## Technical Notes
- `copyPendingToReceipts` downloads and re-uploads each image (Firebase client SDK can't copy Storage objects). For 1-3 images at <2MB each, this is fine.
- The function returns new permanent URLs — use these to update the transaction's `imageUrls` field.
- The CF-side alternative: have `processReceiptScan` copy images to permanent path before writing result. This avoids client-side copy but adds CF execution time. Evaluate trade-off.
- Deferred finding from TD-18-10 review: `copyPendingToReceipts` has no `response.ok` check after `fetch()` — fix this while wiring it up.
- Deferred finding from TD-18-10 review: no per-image timeout on uploads — acceptable risk for now but note for future.
